# Build Request: python-executor

## Overview
A skill that allows me to execute Python scripts on the host machine and capture their output. This enables me to test my own code autonomously before presenting it to Rob.

Core functionality:
- Execute Python scripts by file path
- Capture stdout and stderr in real-time
- Return complete execution results (output, errors, exit code, execution time)
- Timeout protection to prevent hanging scripts
- Working directory support for scripts that need specific paths
- Environment variable support
- Optional arguments to pass to scripts

Use cases:
- Test voice loop scripts before Rob runs them
- Debug syntax errors immediately
- Verify optimization improvements
- Run diagnostic scripts
- Execute utility scripts for file operations
- Test ML model loading and inference scripts

Safety:
- Timeout limit with configurable maximum
- Sandboxed to Python only
- Working directory validation
- Output size limits to prevent memory issues
- Execution logging for audit trail

## Skill Architecture
This is a MiniClaw Discord bot skill. Each skill is a self-contained folder with:
- `handler.js` — The main module. MUST export:
  - `toolDefinition` — An Anthropic tool_use schema object ({ name, description, input_schema })
  - `execute(input)` — An async function that receives the tool input and returns a result object
- `SKILL.md` — Documentation: what the skill does, when to use it, example trigger phrases
- `PROGRESS.md` — Development log (append-only, timestamped entries)
- `data/` — Optional persistent data directory for JSON/text files

## Technical Requirements
- ES modules (import/export, no require())
- Node.js 20+ (no browser APIs)
- The skill runs inside a Discord bot on Windows — no network calls unless explicitly needed
- `process.cwd()` returns the MiniClaw project root
- The handler is loaded via dynamic import: `import('file:///path/to/handler.js')`
- All file operations should use `path.resolve()` for Windows compatibility
- Return objects from execute() — they get JSON-serialized and sent back to Claude API as tool results
- Errors should be returned as `{ success: false, error: 'message' }`, not thrown (unless truly fatal)

## Tool Definition Pattern
```javascript
export const toolDefinition = {
  name: 'python_executor',
  description: 'Clear description of what this tool does and when to use it',
  input_schema: {
    type: 'object',
    properties: {
      action: {
        type: 'string',
        enum: ['action1', 'action2'],
        description: 'What operation to perform'
      }
      // ... more properties
    },
    required: ['action']
  }
};

export async function execute(input) {
  const { action } = input;
  switch (action) {
    case 'action1':
      return { success: true, data: '...' };
    default:
      return { success: false, error: `Unknown action: ${action}` };
  }
}
```

## Specific Requirements
Technical requirements:
- Use Node.js child_process.spawn to execute Python scripts
- Platform-agnostic Windows and Linux support
- Real-time stdout and stderr streaming with buffering
- Proper signal handling for timeouts SIGTERM and SIGKILL
- UTF-8 encoding for output
- Return structured JSON with stdout, stderr, exitCode, executionTime, timedOut flag
- Support both python and python3 executables with auto-detection
- Handle large output gracefully with truncation and warning if needed
- Working directory defaulting to script location
- Optional environment variable merging with process.env

Error handling:
- Script not found gives clear error message
- Python not installed gives clear error message
- Timeout kills process and returns partial output
- Execution errors return stderr with exit code
- Invalid paths are validated before execution

Configuration:
- Default timeout 60 seconds
- Max timeout 300 seconds
- Output size limit 1MB with truncation warning
- Python executable auto-detection
